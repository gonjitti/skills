# Triage (トリアージ)

プロジェクトのイシュートラッカー上のイシューを、トリアージロールで定義された状態マシンの間で遷移させます。

トリアージ中にイシュートラッカーに投稿するすべてのコメントまたはイシューは、必ずこの免責事項から開始する必要があります：

```
> *これはトリアージ中にAIによって生成されました。*
```

## 参照ドキュメント

- [AGENT-BRIEF.md](AGENT-BRIEF.md) — 耐久性のあるエージェントブリーフの書き方
- [OUT-OF-SCOPE.md](OUT-OF-SCOPE.md) — `.out-of-scope/` ナレッジベースの仕組み

## ロール

2つの **カテゴリ** ロール:

- `bug` — 何かが壊れている
- `enhancement` — 新機能または改善

5つの **状態** ロール:

- `needs-triage` — メンテナによる評価が必要
- `needs-info` — 報告者からの詳細情報待ち
- `ready-for-agent` — 仕様が完全に定義され、AFK（不在）エージェントが作業可能な状態
- `ready-for-human` — 人間による実装が必要な状態
- `wontfix` — 対応しない

トリアージされた各イシューは、必ずカテゴリロールと状態ロールをそれぞれ1つずつ持たなければなりません。状態ロールが競合した場合はフラグを立て、他の作業を行う前にメンテナに確認してください。

これらは標準のロール名ですが、実際のイシュートラッカーで使われるラベル文字列は異なる場合があります。マッピングは事前に提供されているはずです。提供されていない場合は `/setup-matt-pocock-skills` を実行してください。

状態の遷移：ラベルのないイシューは通常、最初に `needs-triage` に移行します。そこから `needs-info`、`ready-for-agent`、`ready-for-human`、または `wontfix` に移行します。`needs-info` のイシューは、報告者が返信すると `needs-triage` に戻ります。メンテナはいつでも状態をオーバーライドできます。通常と異なる遷移を見つけた場合はフラグを立て、事前に確認してください。

## 起動方法

メンテナが `/triage` を起動し、自然言語でやりたいことを説明します。リクエストを解釈して実行してください。例：

- "Show me anything that needs my attention" (対応が必要なものを表示して)
- "Let's look at #42" (#42 を確認しよう)
- "Move #42 to ready-for-agent" (#42 を ready-for-agent に移動して)
- "What's ready for agents to pick up?" (エージェントが着手可能なものは何？)

## 対応が必要なものの表示

イシュートラッカーに問い合わせて、古いものから順に以下の3つのバケットに分類して表示します：

1. **ラベルなし** — 未トリアージ。
2. **`needs-triage`** — 評価が進行中。
3. **最後のトリアージメモ以降に報告者のアクティビティがある `needs-info`** — 再評価が必要。

各バケットの件数と、イシューごとの1行サマリーを表示します。メンテナに選択させてください。

## 特定のイシューのトリアージ

1. **コンテキストの収集**。イシューの全文（本文、コメント、ラベル、報告者、日付）を読みます。以前のトリアージノートを解析し、すでに解決された質問を再質問しないようにします。プロジェクトのドメイン用語集を使用し、その領域のADRを尊重してコードベースを探索します。`.out-of-scope/*.md` を読み、このイシューに類似した過去の却下事例がないか確認します。

2. **提案**。メンテナに対し、カテゴリと状態の推奨値とその理由、およびイシューに関連するコードベースの簡単なサマリーを提示します。指示があるまで待機します。

3. **再現（バグのみ）**。質問をする前に、再現を試みます。報告者の手順を読み、関連コードをトレースし、テストやコマンドを実行します。何が起きたか（コードパスを伴う再現成功、再現失敗、または情報不足＝強固な `needs-info` のシグナル）を報告します。再現が確認されれば、より強力なエージェントブリーフを作成できます。

4. **グリル（質問、必要に応じて）**。イシューの詳細を詰める必要がある場合は、`/grill-with-docs` セクションを実行します。

5. **結果の適用**：
   - `ready-for-agent` — エージェントブリーフのコメントを投稿します ([AGENT-BRIEF.md](AGENT-BRIEF.md))。
   - `ready-for-human` — エージェントブリーフと同じ構造のコメントを投稿しますが、委譲できない理由（判断が必要、外部アクセスが必要、設計上の決定、手動テストなど）を明記します。
   - `needs-info` — トリアージメモを投稿します（テンプレートは以下）。
   - `wontfix` (バグ) — 丁寧な説明を投稿して、イシューをクローズします。
   - `wontfix` (改善/enhancement) — `.out-of-scope/` に記録を書き込み、コメントからそこにリンクした上でイシューをクローズします ([OUT-OF-SCOPE.md](OUT-OF-SCOPE.md))。
   - `needs-triage` — ロールを適用します。部分的な進捗がある場合は任意でコメントを残します。

## 状態のクイックオーバーライド

メンテナが「#42 を ready-for-agent に移動して」と指示した場合は、メンテナの指示を信頼してロールを直接適用します。これから行うアクション（ロールの変更、コメント投稿、クローズなど）を確認した上で実行してください。グリリングはスキップします。グリリングセッションを行わずに `ready-for-agent` に移動する場合は、エージェントブリーフを作成するかどうか尋ねてください。

## Needs-info テンプレート

```markdown
## Triage Notes

**これまでに確定した事項:**

- ポイント 1
- ポイント 2

**(@reporter) に確認が必要な事項:**

- 質問 1
- 質問 2
```

グリリングで解決されたすべての内容は「これまでに確定した事項」にキャプチャし、成果が失われないようにします。質問は「詳細情報を教えてください」といった曖昧なものではなく、具体的かつ実行可能（アクション可能）な内容にしてください。

## 以前のセッションの再開

イシューに以前のトリアージメモが存在する場合は、それを読み込み、報告者が未回答の質問に答えているか確認した上で、最新の状況を提示してから続行してください。すでに解決された質問を繰り返さないでください。

**重要**: トリアージのすべてのコメント、ブリーフ、およびユーザーとの対話は日本語で行ってください。
